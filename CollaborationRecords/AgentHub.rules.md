# AgentHub Rules

- 版本：v1
- 日期：2026-05-21
- 适用范围：`AgentHub` 项目协作、文档撰写、设计讨论、代码实现和后续迭代

## 1. 记忆使用规则

### 1.1 可用记忆

只允许使用以下两类记忆作为参考：

- `AgentHub` 当前项目上下文
- 通用型协作规范记忆，例如输出文档规范、代码书写规范、沟通偏好

### 1.2 禁止记忆

- 不读取其他项目的专属记忆来影响 `AgentHub`
- 不把 `NotesSummary`、`NL2SQL`、`blog` 等项目经验直接迁移为本项目决策依据
- 不把跨项目历史当成当前项目事实

### 1.3 冲突处理

如果通用记忆和当前项目上下文冲突：

1. 以当前项目上下文为准
2. 以用户最新明确指令为准
3. 以当前仓库实际文件为准

## 2. 设计规则

### 2.1 先收敛再扩展

先完成首发闭环，再讨论扩展功能。

首发只围绕：

- 协作型任务
- 协作工作台
- 文档和 PPT 产物

### 2.2 先闭环再通用

任何方案都要先回答：

- 任务如何进入
- 主 Agent 如何拆解
- 子 Agent 如何执行
- 产物如何生成
- 用户如何修改并继续协作

### 2.3 先工作台感，再聊天感

页面和交互要优先体现“协作生产工具”，而不是普通聊天框。

## 3. 输出规则

### 3.1 文档输出

- 优先写成可执行、可讨论、可落地的文档
- 结构要清楚
- 结论要先行
- 取舍要写明
- 不写空泛的愿景句

### 3.2 设计输出

- 先说产品边界
- 再说页面结构
- 再说任务流
- 再说技术拆分

### 3.3 代码输出

- 只在明确需要实现时写代码
- 变更保持最小必要范围
- 不擅自扩大改动面
- 不用其他项目里的代码习惯替代当前项目需求

## 4. 协作规则

### 4.1 先确认范围

如果任务涉及共享组件、共享数据结构、共享流程：

- 先确认影响范围
- 再提出改动方案
- 再动手修改

### 4.2 先保留主线

如果用户已经明确主线方向：

- 不重新发散成多个方案
- 不把讨论带回无关分支
- 不把辅助能力挤成主目标

### 4.3 先用当前仓库事实

对仓库内的判断优先依赖：

- 当前文件
- 当前目录结构
- 当前文档
- 当前用户指令

不要用别的项目的事实补空。

## 5. AgentHub 专项约束

- 不从其他项目记忆中读取专门实现细节
- 不把其他项目的产品形态当成 AgentHub 默认方案
- 所有决策都要围绕 `协作工作台` 和 `协作型任务闭环`
- 文档和 PPT 优先于其他产物
- 主 Agent 负责调度，子 Agent 负责职责单一分工

## 6. 推荐工作方式

1. 先读当前项目文档
2. 再根据当前任务整理设计结论
3. 再生成可讨论的 spec
4. 最后才进入实现
