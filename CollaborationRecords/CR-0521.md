# CR-0521 | AgentHub 多 Agent 协作平台首发方案记录

- 日期：2026-05-21
- 记录目标：基于 `otherfiles/智能纪要：【AgentHub-多agent协作平台】课题讲解 2026年5月20日.pdf` 的会议内容，结合当前讨论结果，整理项目首发方向、MVP 拆分、页面结构和任务流设计。

## 1. 结论先行

本项目首发不做通用聊天产品，而是做一个以 `协作工作台` 为核心的 `协作型任务闭环`。

首发重点如下：

- 任务闭环类型：协作型任务
- 产品形态：协作工作台
- 优先产物：文档、PPT
- 核心能力：主 Agent 负责拆解任务，多个子 Agent 并行协作，最终汇总成可预览、可编辑、可继续追问的产物

这一定义与会议纪要中的要求一致：

- 以 IM 聊天为核心交互范式
- 需要主 Agent 作为协调器
- 要支持多 Agent 调度、失败降级、冲突处理
- 要有产物预览、编辑、二次交互能力
- 交付物要能体现 AI 协作记录

## 2. 项目定位

### 2.1 设计目标

将用户的一个复杂内容任务，转化成一条完整的协作生产链路：

1. 用户发起需求
2. 主 Agent 理解目标并拆解任务
3. 多个子 Agent 并行完成子任务
4. 系统汇总为文档和 PPT 产物
5. 用户在工作台内直接修改、追问、重分发任务

### 2.2 产品表达重点

首发版本要让评委看到的不是“我们也做了一个聊天框”，而是：

- 任务是如何被协作拆解的
- 多个 Agent 是如何并行工作的
- 产物是如何从消息流中长出来的
- 用户是如何在同一个工作台里继续推进协作的

### 2.3 首发边界

为了控制交付风险，首发不优先做以下内容：

- 不先追求完整多端
- 不先追求复杂的通用编辑器
- 不先追求过多 Agent 平台接入
- 不先追求所有产物类型都支持

首发只抓一个最能讲清楚的闭环：`协作型任务 -> 文档/PPT 产物 -> 预览/修改 -> 二次协作`

## 3. 拆分一：MVP 功能清单

### 3.1 必做功能

#### 3.1.1 任务输入与会话入口

- 用户可以发起一个协作型任务
- 支持自然语言描述需求
- 支持把任务挂到一个会话中继续推进

#### 3.1.2 主 Agent 任务规划

- 主 Agent 负责理解用户意图
- 主 Agent 输出任务拆解结果
- 主 Agent 把复杂需求拆成多个可执行子任务
- 主 Agent 负责协调子 Agent 的执行顺序与并行关系

#### 3.1.3 子 Agent 并行协作

- 至少支持两个子 Agent 并行工作
- 每个子 Agent 负责一个明确子目标
- 子 Agent 的状态要能在界面中可见
- 支持任务完成、失败、重试、降级等基础状态

#### 3.1.4 文档产物生成与预览

- 支持生成结构化文档
- 支持文档预览
- 支持文档局部修改
- 支持针对某一段内容继续追问或重生成

#### 3.1.5 PPT 产物生成与预览

- 支持生成 PPT 大纲或逐页内容
- 支持页面列表预览
- 支持单页重新生成
- 支持按页查看标题、要点和顺序

#### 3.1.6 协作记录沉淀

- 记录任务拆解过程
- 记录 Agent 协作过程
- 记录用户修改与再追问过程
- 为后续 `spec / skill / rules` 这类协作规范沉淀提供内容

### 3.2 可延后功能

以下内容可作为第二阶段增强，不影响首发闭环：

- 完整的代码编辑器
- 网页产物深度预览
- 更复杂的版本对比
- 多端同步
- 更完整的 Agent 市场
- 用户自定义 Agent 的高级配置
- 复杂的权限与团队管理

### 3.3 暂不做功能

为了避免首发发散，以下能力先不做：

- 通用型全量创作平台
- 所有产物类型一开始全覆盖
- 复杂的多人实时协同编辑
- 太重的流程配置系统
- 过多无差异化的装饰性功能

## 4. 拆分二：页面结构

### 4.1 总体布局

建议首发 UI 直接定位为 `协作工作台`，而不是传统 IM。

页面建议分为三栏：

- 左侧：会话与 Agent 导航
- 中间：协作聊天与任务流
- 右侧：产物预览与编辑

### 4.2 左侧导航区

左侧区域承担“找到任务与找到协作者”的作用：

- 会话列表
- 当前任务列表
- Agent 列表
- 常用能力标签

目标是让用户快速知道：

- 现在在哪个任务里
- 谁在参与协作
- 当前任务属于什么类型

### 4.3 中间协作区

中间是整个工作台的主区域，承载任务推进过程：

- 用户消息
- 主 Agent 规划消息
- 子 Agent 回传消息
- 任务拆解卡片
- 执行状态卡片
- 失败 / 重试 / 降级提示

这个区域的核心不是“聊天记录”，而是“协作过程可视化”。

### 4.4 右侧产物区

右侧用于展示协作结果：

- 文档预览
- PPT 预览
- 版本历史
- 局部修改入口
- 重新生成入口

如果后续扩展能力允许，这一栏也可以继续承接：

- diff 视图
- 引用片段
- 产物说明

### 4.5 推荐的页面信息层级

建议优先级如下：

1. 当前任务是否在推进
2. 主 Agent 是否完成拆解
3. 子 Agent 是否并行执行
4. 产物是否已生成
5. 用户是否能直接修改并继续协作

这五层优先级比单纯展示聊天气泡更重要。

## 5. 拆分三：任务流设计

### 5.1 标准任务流

首发版本建议固定成下面的协作链路：

1. 用户输入一个协作型任务
2. 主 Agent 识别任务目标
3. 主 Agent 拆分子任务
4. 多个子 Agent 并行执行
5. 系统汇总结果
6. 生成文档与 PPT 产物
7. 用户在工作台中修改或追问
8. 主 Agent 根据新上下文继续分发任务

### 5.2 主 Agent 职责

主 Agent 不是普通回答器，而是协作协调器，负责：

- 理解用户目标
- 判断任务复杂度
- 拆分任务边界
- 分配给不同子 Agent
- 聚合结果
- 处理失败与冲突
- 维护上下文连续性

### 5.3 子 Agent 职责

子 Agent 要尽量职责单一，避免一个 Agent 什么都干：

- 文档 Agent：负责章节结构、正文组织、总结提炼
- PPT Agent：负责页面结构、页标题、要点拆分、演示表达
- 补充 Agent：负责风险点、亮点、细节优化、措辞润色

### 5.4 协作状态设计

建议在任务流里至少定义这些状态：

- 待拆解
- 已拆解
- 并行执行中
- 部分完成
- 已汇总
- 已生成产物
- 用户已修改
- 再次协作中

这套状态足够支撑首发闭环，也便于后面扩展。

### 5.5 失败与冲突处理

首发不必把复杂冲突处理做深，但需要有可见策略：

- 某个子 Agent 失败时，主 Agent 可以重试或替换执行
- 多个结果冲突时，主 Agent 负责归并或择优
- 若产物局部有问题，允许用户指定局部重生成

这部分是评审时很容易被问到的，因此至少要有“机制存在”的展示。

## 6. 首发演示建议

建议 Demo 主题固定为一个内容制作任务，例如：

- 用户要求生成一份 `AgentHub 协作平台介绍文档 + 路演 PPT`

演示时按以下步骤走：

1. 用户发起需求
2. 主 Agent 拆成文档和 PPT 两条协作分支
3. 子 Agent 并行生成内容
4. 工作台展示状态卡片和回传结果
5. 右侧同步显示文档和 PPT 产物
6. 用户修改某一段后再次发起协作
7. 展示版本更新与二次交互

这个场景能同时覆盖：

- 协作能力
- 产物能力
- 工作台感
- 二次编辑与再协作能力

## 7. 当前阶段的产品判断

### 7.1 为什么选协作型任务

因为它最能体现这次题目的核心要求：

- 多 Agent
- 主 Agent 调度
- 产物协作
- 工作台式体验

### 7.2 为什么选协作工作台

因为协作工作台比单纯聊天更能承接复杂任务，也更容易让用户感知到“这是一个生产工具，而不是一个聊天壳子”。

### 7.3 为什么优先文档和 PPT

因为这两类产物最适合首发展示：

- 文档适合表达结构和内容组织
- PPT 适合表达展示和汇报结果
- 两者都容易形成完整 Demo 闭环
- 两者都能体现“生成 -> 预览 -> 修改 -> 再协作”

## 8. 下一步建议

后续建议继续按下面顺序推进：

1. 先定首发信息架构图
2. 再定主 Agent 与子 Agent 的最小接口
3. 再定文档和 PPT 的产物数据结构
4. 最后补协作记录规范，形成 `spec / skill / rules`

---

这份记录对应当前已达成的设计结论：

- 协作型任务作为首发闭环
- 协作工作台作为产品形态
- 文档和 PPT 作为首批产物
- 主 Agent 负责调度，子 Agent 负责分工
- 页面强调任务流和产物流，而不是纯聊天
