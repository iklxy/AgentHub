## 1. 开发目标

### 1.1 首发目标

首发版本只实现一条可闭环的协作链路：

1. 用户在首页进入全新的对话列表，支持新建、置顶、归档和搜索
2. 支持单聊模式与群聊模式，分别对应 1v1 和多 Agent 协作
3. 主 Agent / Orchestrator 在群聊模式下理解意图、拆解任务、并行分派，所有主 Agent / 子 Agent 都是外部接入平台的包装器
4. 消息流支持文本、代码块、图片、文件附件、网页预览卡片、Diff 视图卡片、部署状态卡片
5. 消息支持回复、引用、重新生成、复制代码、一键应用 Diff、展开预览
6. 聊天历史自动进入上下文，支持 pin 关键消息作为长期上下文
7. 产物以内联卡片、全屏预览、代码编辑器和版本化内容的形式回流到对话流和右侧产物区
8. 用户可以在二级入口查看某个 Agent 的工作记录和 rules，并单独调整
9. 部署发布作为 P2 能力，通过聊天指令触发并返回部署状态卡片

### 1.2 首发边界

首发不做：

- 复杂多人实时协同编辑
- 全量通用创作平台
- 完整多端同步
- 过重的流程配置系统
- 过多无差异化装饰能力

首发只保留：

- 首页对话列表
- 任务详情 / 协作工作台
- 单聊 / 群聊模式
- 文档、PPT、网页预览、代码预览产物
- Agent 协作记录
- Agent 级 rules 编辑
- 部署发布 P2 能力
- 主/子 Agent 由外部平台接入后包装成系统内的 Agent 实体

## 2. 技术选型

### 2.1 总体方案

推荐采用 `Web 前端 + Golang 微服务后端 + Python Agent Workflow + PostgreSQL + Redis + 对象存储` 的组合。

这样选的原因：

- 前端需要较强的状态管理和复杂交互能力，TypeScript 适配最佳
- 核心业务 API 需要高并发、强类型和清晰服务边界，Go 更适合做微服务底座
- Agent 工作流大量依赖 LLM SDK、异步任务、数据解析和 prompt 处理，Python 生态更直接
- PostgreSQL 足以承载结构化任务、消息、规则、版本和审计数据
- Redis 适合做队列、事件缓存、任务状态加速和 SSE 推送缓冲
- 对象存储用于保存文档 / PPT 导出文件、附件和大文件快照

### 2.2 前端技术栈

- 语言：TypeScript
- 框架：Next.js 15
- 运行时：React 19
- 样式：Tailwind CSS
- 组件：shadcn/ui + Radix UI
- 状态管理：Zustand
- 请求管理：TanStack Query
- 富文本编辑：Tiptap
- PPT 预览：React 结构化渲染组件
- 流式事件接收：SSE
- 图表 / 流程图：Mermaid 或自定义节点图

### 2.3 后端技术栈

- 语言：Go 1.22+
- Web 框架：Gin 或 Fiber
- API 协议：REST + OpenAPI
- 内部服务通信：gRPC + Protobuf
- 配置管理：Viper
- 数据库访问：sqlc / ent / GORM 三选一，优先 sqlc 或 ent
- 数据库迁移：golang-migrate
- 缓存 / 队列：Redis
- 文件存储：S3 兼容对象存储，优先 MinIO 便于本地开发
- 日志：zap 或 zerolog
- 链路追踪：OpenTelemetry

### 2.4 Agent 工作栈

- 语言：Python 3.12
- Agent 编排：Python Workflow Worker
- 任务调度：Redis Queue / RabbitMQ / Temporal 三选一，首发优先 Redis Queue 或 Temporal
- Agent prompt 载体：Markdown + YAML
- Agent 输出协议：JSON Schema
- Agent 工具调用：结构化 tool schema
- 规则执行：规则编译器将 Markdown rules 转成 runtime JSON
- 运行边界：Python 不承担核心业务 CRUD，仅承担 Agent 推理、编排、解析、重试、降级
- 统一适配：支持 Claude Code、Codex、OpenCode 等外部 Agent 平台，也支持用户自建 Agent
- 角色定位：系统内的主 Agent / 子 Agent 都是外部平台 Agent 的包装实例，本地只保存映射、调度和结果归一化逻辑

### 2.5 存储与协议

- 主数据库：PostgreSQL 16
- 关键字段类型：JSONB、TIMESTAMP WITH TIME ZONE、UUID
- 搜索：PostgreSQL FTS + GIN 索引
- 事件协议：REST + SSE
- 文件协议：S3 API
- 内部 Worker 协议：gRPC / Redis queue / 事件总线

## 3. 语言分工

### 3.1 前端语言

前端全部使用 TypeScript。

原因：

- 任务、Agent、规则、产物的结构复杂，TS 适合做强类型约束
- UI 组件和数据 DTO 可以共享类型定义
- 后续若加 WebSocket、复杂表单和 schema 编辑，TS 维护成本更低

### 3.2 后端语言

后端使用 Go。

原因：

- 首发需要把系统拆成多个微服务，Go 对服务边界、并发和部署更友好
- API 层、权限层、任务层、产物层、规则层更适合用强类型语言沉淀契约
- Go 更适合作为高并发工作台的后端基础设施

### 3.3 Agent 工作语言

Agent 工作语言同样使用 Python。

原因：

- 可以直接复用后端的数据模型、校验逻辑和工具函数
- 子 Agent 的输出解析、重试、上下文整理更简单
- 主 Agent 调度和执行状态追踪可以统一在 Python Orchestrator & Agent Wrapper Layer 内完成

### 3.4 规则与提示词语言

- 规则文案：Markdown
- 结构化配置：YAML / JSON
- 运行时协议：JSON Schema

说明：

- Markdown 适合人工编辑 rules
- YAML 适合写 prompt 模板和参数配置
- JSON Schema 适合校验 Agent 输出

### 4.2 服务职责

#### 前端

- 负责展示首页对话列表
- 负责任务详情页三栏工作台
- 负责文档和 PPT 预览、编辑、版本切换
- 负责 Agent 记录和 rules 编辑
- 负责接收 SSE 事件并实时更新 UI

#### API Gateway 与业务微服务（Go）

- API Gateway 负责统一鉴权、路由、限流、聚合查询和 SSE 输出
- 各业务微服务负责各自领域的 CRUD 与领域规则
- Task Service 负责任务主表和任务状态机
- Conversation Service 负责对话列表、单聊/群聊模式、置顶、归档、搜索和活跃排序
- Message Service 负责消息流
- Artifact Service 负责文档和 PPT 产物
- Rules Service 负责 Agent rules 与版本
- Agent Registry Service 负责 Agent 定义、接入与能力标签
- Agent Adapter Service 负责 Claude Code / Codex / OpenCode 等统一适配、外部 Agent 的创建/同步/调用、输出归一化和回调处理
- Deployment Service 负责部署发布、预览 URL、静态站点、容器化和源码包
- File Service 负责文件上传、预览和导出
- Event Gateway 负责将领域事件转成前端可消费的 SSE 流

#### 领域微服务之间的通信

- 同步调用：gRPC
- 异步事件：Event Bus 或 Redis Stream
- 只读聚合：API Gateway 聚合多个服务结果后返回前端

#### Python Agent Workflow Layer

- 负责将内部 Agent 角色包装到外部平台 Agent 实例上
- 负责执行主 Agent 拆解的编排逻辑
- 负责执行子 Agent 产出的编排逻辑
- 负责重试、降级和冲突归并
- 负责把结果整理成可落库的标准结构
- 负责单聊模式下的直接执行和群聊模式下的 Orchestrator 聚合
- 负责对话历史、pin 消息和临时上下文拼装

#### 数据层

- PostgreSQL 保存结构化业务数据和版本
- Redis 保存队列、事件缓冲和热点状态
- 对象存储保存导出文件和大附件

用途：定义 Agent 实体，包含主 Agent、文档 Agent、PPT Agent 和后续自定义 Agent。

- `avatar_url` TEXT

### 6.7 `agent_connectors`

用途：承载外部 Agent 平台接入配置，支持外部平台 SDK/API、CLI Bridge 和自定义 Webhook。

`connector_type` 建议枚举为 `claude_code`、`codex`、`opencode`、`custom_webhook`、`internal_python`。

说明：

- 一个 connector 代表一种外部平台接入方式或一套认证配置
- `config_json` 保存平台 endpoint、token、project、workspace、model mapping 等非敏感参数
- 如果平台本身支持 Agent 创建 API，则 connector 可以在同步时生成远端 Agent
- 如果平台不支持 Agent 创建 API，则 connector 只负责把本地 Agent wrapper 路由到对应的 webhook / CLI / SDK 执行器
- `internal_python` 仅作为本地测试、工具型辅助 Agent 或兜底执行器的保留选项，不作为首发主/子 Agent 的默认实现方式

字段建议：

## 7. API 设计

### 7.1 接口风格

建议采用：

- REST 作为主接口
- SSE 作为实时事件通道
- JSON 作为默认数据格式
- Cursor 分页作为列表分页方式

### 7.2 统一响应格式

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

```json
  "message": "task not found",
```

## 8. Agent 工作协议

### 8.1 主 Agent 职责

主 Agent 负责：

- 理解用户目标
- 拆解任务
- 调度子 Agent
- 合并子 Agent 结果
- 处理失败和冲突
- 生成最终可交付内容结构
- 在群聊模式下充当 Orchestrator，识别 @ 指定、自动分派和并行执行
- 汇总多个 Agent 的回复并在聊天流中输出结果卡片
- 处理代码冲突、结果冲突和执行降级

### 8.2 子 Agent 职责

- 文档 Agent：章节结构、正文组织、总结提炼
- PPT Agent：页面结构、页标题、要点拆分、演示表达
- 补充 Agent：风险点、亮点、措辞优化、细节补全
- 外部平台 Agent：通过统一 adapter 接入 Claude Code、Codex、OpenCode 等平台

### 8.3 对话模式规则

- 单聊模式：1v1 对单个 Agent 讲话，适合明确任务
- 群聊模式：一个对话中包含多个 Agent，可通过 @ 指定或由 Orchestrator 自动分派
- 对话历史默认作为上下文输入
- pin 的消息会作为长期上下文在后续轮次中优先注入

### 8.4 外部 Agent 接入流程

系统里的主 Agent 和子 Agent 本质上都是外部平台 Agent 的包装实例，接入流程建议如下：

1. 在 `agent_connectors` 中创建一个外部平台连接器，配置鉴权、endpoint、模型映射和能力信息
2. 在 `agents` 中创建一个本地 wrapper 记录，写入 `connector_id`、`provider_type` 和 `remote_agent_id`
3. 如果外部平台支持 Agent 创建 API，则调用 `POST /api/agents/{agent_id}/sync` 在外部平台生成或更新远端 Agent
4. 如果外部平台不支持创建 API，则把本地 wrapper 绑定到 webhook、CLI Bridge 或 SDK 执行器
5. 会话参与者表 `session_participants` 只记录本地 wrapper Agent，不直接暴露外部平台内部对象
6. Orchestrator 根据用户消息、@ 指定、规则和会话状态选择要调用的 wrapper Agent
7. `agent-adapter-service` 将内部统一消息协议翻译成对应平台的调用格式
8. 外部平台返回流式输出或最终结果后，adapter 做结果归一化，再写回 `messages`、`task_step_runs`、`artifact_versions` 或 `deployments`
9. 如果平台支持回调，则通过 webhook 触发结果回写；如果不支持，则通过轮询或长连接获取结果
10. 所有运行记录都保留本地视图，外部平台只作为执行后端

### 8.5 Agent 输出协议

每次 Agent run 的结果必须符合统一 JSON 结构，便于落库和回放。

```json
{
  "run_id": "uuid",
  "agent_id": "uuid",
  "task_id": "uuid",
  "step_id": "uuid",
  "status": "succeeded",
  "summary": "输出了一版技术选型章节",
  "deliverables": [
    {
      "type": "artifact_patch",
      "artifact_type": "document",
      "target": "section:3",
      "content": {}
    }
  ],
  "issues": [],
  "confidence": 0.86,
  "usage": {
    "input_tokens": 0,
    "output_tokens": 0
  }
}
```

### 8.6 规则编译流程

运行时读取规则的顺序建议如下：

1. 任务临时 rules
2. 当前 Agent rules
3. 项目级默认 rules
4. 通用协作规范

编译后的规则不直接暴露给用户，而是作为 Agent 的运行上下文。

### 8.7 重试与降级

- 步骤失败后允许重试
- 多次失败后由主 Agent 降级处理
- 降级可选择缩小目标、简化结构、只保留大纲
- 所有失败和降级必须写回 `task_step_runs` 和 `audit_events`

## 10. 文档与 PPT 生成策略

### 10.1 文档

建议文档以 `Markdown + 富文本块结构` 双存储。

原因：

- Markdown 便于 Agent 生成和编辑
- 块结构便于前端编辑器渲染
- 后续导出 DOCX 也方便

文档导出建议：

- 首发可先支持 Markdown / HTML 预览
- 后续再接 `python-docx` 生成 DOCX
- Agent 回复中可内联文档预览卡片，点击后展开全屏文档编辑器

### 10.2 PPT

建议 PPT 以结构化 slide JSON 为主。

原因：

- 每页结构清晰
- 便于局部重生成
- 便于版本对比
- 便于前端按页预览

PPT 导出建议：

- 首发可以先做网页预览
- 后续再接 `.pptx` 导出
- PPT 浏览作为 P2 能力，优先作为产物卡片中的展开预览而不是主编辑器

### 10.3 网页与代码

网页与代码预览建议以统一的产物卡片承载：

- 网页产物通过 iframe / WebView 预览
- 代码产物通过代码编辑器和 Diff 视图预览
- 点击产物卡片后可进入全屏预览或全屏编辑器
- 对话式局部修改支持“选中代码 -> 在聊天中描述修改 -> 生成 Diff -> 一键应用”

### 10.4 部署发布（P2）

部署发布在首发中按 P2 设计，但数据结构和接口需要预留：

- 聊天中可直接输入“部署”
- Agent 返回部署状态卡片
- 可生成预览 URL
- 可触发静态站点部署
- 可触发容器化部署
- 可生成源码打包下载

### 10.5 局部修改

文档和 PPT 都要支持“只修改局部内容”，因此版本写入需要保留：

- 修改位置
- 修改原因
- 影响范围
- 来源 run

## 11. 开发语言与仓库建议结构

### 11.1 推荐语言组合

- 前端：TypeScript
- 后端：Go
- Agent Workflow：Python
- 规则/提示词：Markdown + YAML

### 11.2 推荐目录结构

```text
AgentHub/
├─ apps/
│  ├─ web/           # Next.js 前端
```

## 12. 风险点与处理建议

### 12.1 Agent 输出不稳定

处理方式：

- 强制 JSON Schema 输出
- 每次 run 都做结构化校验
- 校验失败进入重试或降级

### 12.2 文档和 PPT 内容漂移

处理方式：

- 版本化存储
- artifact 以 task 为主键聚合
- 每次重生成记录来源 version

### 12.3 rules 互相覆盖

处理方式：

- 明确优先级
- 规则入库时写 scope
- 运行时编译后再下发给 Agent

### 12.4 外部 Agent 平台接入不一致

处理方式：

- 统一 connector 抽象
- 平台差异只在 connector 层吸收
- 对上游暴露统一 run / result 协议

## 14. 结论

首发开发方案的核心取舍是：

- 用 TypeScript 做前端工作台
- 用 Go 做核心后端微服务
- 用 Python 做 Agent Workflow
- 用 PostgreSQL 存结构化任务和版本
- 用 Redis 做队列和事件
- 用 SSE 做实时状态刷新
- 用 OpenAPI + Protobuf 管理接口契约
- 用 Markdown / YAML 做 rules 和 prompts
- 用 `Agent` 作为可调优实体，而不是只把协作记录当日志

这套方案的目标不是先做最全，而是先做最稳的协作闭环。
