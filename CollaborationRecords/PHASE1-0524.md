## 1. 阶段目标

第 1 阶段的目标不是把所有能力做全，而是做出一条真正可跑通的核心链路：

1. 首页进入对话列表
2. 用户创建对话，选择单聊或群聊
3. 系统通过外部 `Agent Connector` 接入主 Agent 和子 Agent
4. 用户在对话中发消息，Orchestrator 负责理解意图、分派 Agent、聚合结果
5. Agent 返回文本、代码、网页、文档、PPT、Diff、部署状态等结构化消息
6. 用户可以在同一上下文里继续追问、引用、重生成、复制代码、应用 Diff、pin 关键消息
7. 对话内的文档 / PPT / 网页 / 代码产物可预览，可局部编辑，可版本化
8. 协作记录页可以查看单个 Agent 的工作记录与 rules，并支持单 Agent 级规则调整

第 1 阶段必须保证：

- 主 Agent 和子 Agent 都是外部平台 Agent 的包装实例
- `Agent Connector` 接入链路可用
- 单聊与群聊都能跑通
- 聊天上下文和 pin 机制可用
- 产物预览卡片与消息操作可用

## 3. 核心依赖

第 1 阶段之所以必须先接入外部 `Agent Connector`，原因是整个链路依赖真实外部执行后端：

- Orchestrator 需要把消息发给外部 Agent 平台
- 群聊模式需要多个外部 Agent 协同
- 结果回传必须经过 adapter 归一化
- 主 Agent / 子 Agent 在系统内只是包装实例，不在本地执行真正推理

因此第 1 阶段的优先级应该是：

1. 连接器可用
2. Agent 包装器可用
3. 会话与消息可用
4. Orchestrator 可用
5. 产物卡片可用
6. 规则可调优

## 5. 模块拆解

### 5.1 Connector 模块

目标：把外部 Agent 平台统一接入到系统中。

任务：

- 定义 connector schema
- 支持 `claude_code / codex / opencode / custom_webhook`
- 支持连接测试
- 支持同步远端 Agent
- 支持本地 wrapper 绑定 connector
- 支持外部平台执行结果回调或轮询拉取

交付：

- `agent_connectors` 表
- `GET /api/connectors`
- `POST /api/connectors`
- `PATCH /api/connectors/{connector_id}`
- `POST /api/connectors/{connector_id}/test`
- `POST /api/agents/{agent_id}/sync`

### 5.2 Agent Wrapper 模块

## 7. API 开发顺序

建议按以下顺序实现：

1. 连接器 API
2. Agent wrapper API
3. 会话 API
4. 消息 API
5. Orchestrator dispatch API
6. 产物 API
7. rules API
8. SSE 事件 API

这样排的原因是：

- 先接入外部 Agent，才有执行后端
- 先有会话和消息，才能调试对话链路
- 先有 Orchestrator，才能串起单聊/群聊
- 产物和 rules 放在后面可以减少前期阻塞

## 9. Orchestrator 开发顺序

### 9.1 单聊链路

1. 读取会话历史
2. 拼装上下文
3. 调用单个外部 Agent wrapper
4. 接收结果并写回消息
5. 如有产物 patch，同步更新 artifact 版本

### 9.2 群聊链路

1. 识别用户意图
2. 选择候选 Agent
3. 根据规则和能力标签分派
4. 并行调用多个 Agent wrapper
5. 聚合回复
6. 输出结果卡片
7. 更新步骤、产物和审计记录

### 9.3 降级与冲突

- 单 Agent 失败可重试
- 多 Agent 冲突由主 Agent 汇总
- 如果远端平台不可用，进入降级模式，只输出大纲或简化结果

## 11. 验收标准

第 1 阶段完成时，至少要满足：

- 能在前端创建 connector 并测试连通性
- 能把一个 Agent wrapper 绑定到某个外部平台
- 能创建一个单聊对话并发送多轮消息
- 能创建一个群聊对话并 @ 或自动分派多个 Agent
- 能看到 Agent 的回复、产物卡片和状态卡片
- 能 pin 消息作为长期上下文
- 能在协作记录页修改某个 Agent 的 rules
- 能把外部平台返回结果写回本地消息和版本表

## 12. 主要风险

### 12.1 外部 Agent 平台能力不一致

- 风险：不同平台的 SDK、Webhook、CLI 和输出格式不统一
- 对策：adapter 层统一协议，平台差异只留在 connector 内

### 12.2 Orchestrator 上下文失控

- 风险：多轮对话和 pin 消息会让上下文膨胀
- 对策：上下文分层、摘要化、只注入必要 pin

### 12.3 产物和消息不同步

- 风险：Agent 先回了消息，但产物版本没及时落库
- 对策：所有产物 patch 必须先落版本，再回写消息引用

### 12.4 第 1 阶段范围过大

- 风险：把部署、多端、复杂编辑全部塞进来会拖慢首发
- 对策：部署只保留表结构和接口壳，作为 P2

## 13. 推荐执行顺序

最推荐的执行顺序是：

1. `agent_connectors` 和 `agents` 的 wrapper/connector 逻辑
2. `task_sessions`、`session_participants`、`messages`
3. Orchestrator dispatch
4. 产物容器和预览卡
5. rules 编辑与 pin 上下文
6. SSE 和协作记录页

这条顺序的核心原则是：

- 先让外部 Agent 接进来
- 再让消息流跑起来
- 再让对话和产物闭环起来
- 最后再优化展示和稳定性
