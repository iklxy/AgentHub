## 1. v0.1 定位

v0.1 不做完整的多 Agent 协作平台，而是先做 `AgentHub` 的最小可运行闭环，用来验证下面这条主线：

1. 用户注册 / 登录
2. 进入自己的 workspace
3. 创建 task
4. 在 task 内和主 Agent `银河` 对话
5. 后端通过 `Claude Code CLI` 执行用户请求
6. 将 Agent 返回内容以聊天消息的形式展示在页面中

这个版本的目标不是证明“多 Agent 已经完成”，而是证明下面三件事已经成立：

- AgentHub 的核心交互确实是 IM 风格聊天
- 一个 task 可以作为独立协作工作单元存在
- 主 Agent 可以被真实调用并把结果回流到消息流中

## 2. v0.1 范围

### 2.1 必做

- 注册页
- 登录页
- workspace 首页
- workspace 名称与描述的初始化填写和二次编辑
- task 创建
- task 列表展示
- task 详情页
- 默认主 Agent `银河`
- 用户与 `银河` 的基础对话
- 后端通过 `Claude Code CLI` 获取回复
- 消息列表实时刷新或发送后刷新

### 2.2 暂不做

- 多 Agent 并行协作
- Agent rules 配置页
- 协作记录页
- 文档 / PPT 产物预览区
- 文件上传
- Diff 卡片
- 网页预览卡片
- 群聊模式
- 复杂任务编排
- 流式输出
- 持久化的 Agent 执行日志面板

### 2.3 设计原则

- 首页首先是 `workspace + task` 入口，不是普通聊天首页
- 一个 task 对应一个独立对话上下文
- v0.1 虽然只有一个主 Agent，但 UI 结构要为后续 Agent 扩展留位置
- 先保证链路跑通，再考虑多 Agent、产物区、规则系统

## 3. 页面说明

## 页面切换和点击UI、触碰UI的时候，加入一些动态的效果

### 3.1 注册页

目标：完成用户注册。

页面定位：

- 注册页是一个轻量入口页
- 视觉上要简洁、干净、可信，不要做成营销落地页
- 页面重点是让用户快速完成注册，而不是承载复杂介绍信息

展示结构建议：

- 左侧或上方：产品名称 `AgentHub`
- 一句简短说明：例如“多 Agent 协作工作台”
- 中间主体：注册表单卡片
- 底部辅助区：跳转登录入口

注册表单卡片建议包含：

- 标题：注册 AgentHub
- 副标题：创建你的协作工作区
- 用户名输入框
- 邮箱输入框
- 密码输入框
- 确认密码输入框
- 注册按钮
- 去登录的文本按钮

视觉建议：

- 卡片宽度控制在中等范围，避免表单过宽
- 输入框、按钮、提示文案保持统一圆角和统一间距
- 主按钮要足够明显，次级跳转入口弱化处理
- 浅色背景下可用少量图形或渐变提升识别度，但不能抢表单主体

字段建议：

- 用户名
- 邮箱
- 密码
- 确认密码

交互要求：

- 注册成功后跳转登录页，或直接登录后跳转首页，二选一即可，v0.1 推荐直接跳转登录页
- 需要有基础错误提示，例如邮箱已存在、密码不一致、参数为空
- 提交中按钮进入 `loading` 状态，避免重复提交
- 输入框校验建议在失焦时给出即时提示

状态展示建议：

- 默认态：正常输入
- 校验错误态：输入框红色描边 + 错误文本
- 提交中态：按钮显示加载中
- 成功态：短暂提示后跳转
- 接口失败态：在表单顶部或按钮上方给出集中错误提示

接口：

- `POST /api/register`

### 3.2 登录页

目标：完成用户登录并建立会话。

页面定位：

- 登录页风格应与注册页保持一致
- 让用户默认感受到“进入工作台”，而不是“进入社交聊天应用”

展示结构建议：

- 产品名称区
- 登录表单卡片
- 去注册入口

登录表单卡片建议包含：

- 标题：登录 AgentHub
- 副标题：进入你的协作工作区
- 邮箱输入框
- 密码输入框
- 登录按钮
- 去注册的文本按钮

视觉建议：

- 注册页和登录页建议共用一个 `AuthLayout`
- 卡片区、背景、标题、按钮样式保持统一
- 页面主视觉不要过重，重点仍然是表单和进入路径清晰

字段建议：

- 邮箱
- 密码

交互要求：

- 登录成功后跳转 workspace 首页
- 登录失败时展示明确错误信息
- 登录按钮需要有加载态
- 若用户已登录，可直接重定向到 workspace 首页

状态展示建议：

- 默认态：展示登录表单
- 登录失败态：表单顶部展示错误提示
- 登录中态：按钮禁用
- 已登录重定向态：页面不再展示表单，直接跳转

接口：

- `POST /api/login`

说明：

- 原始草稿里登录和注册接口描述写反了，这里已经纠正：
  - `POST /api/register` 用于注册
  - `POST /api/login` 用于登录

### 3.3 Workspace 首页

目标：承接用户的 task 管理入口。

页面定位：

- 这是用户登录后的第一个主页面
- 首页应该像一个 `workspace / task hub`
- 它的核心不是聊天本身，而是让用户管理和进入 task

页面结构建议：

- 左侧固定栏：workspace 信息 + task 操作区 + task 列表
- 右侧主区域：空态说明或 task 概览提示

推荐宽度关系：

- 左侧栏较窄，承担导航和管理功能
- 右侧区域较宽，承担说明与内容承接功能

#### 3.3.1 左侧固定栏

建议自上而下分成 4 个区块：

1. workspace 头部区
2. workspace 描述区
3. task 操作区
4. 当前用户信息区

##### workspace 头部区

展示内容：

- workspace 名称
- 编辑按钮

展示要求：

- 名称是左栏最强视觉层级
- 如果名称为空，显示默认占位文案，例如“未命名工作区”

##### workspace 描述区

展示内容：

- workspace description

展示要求：

- 描述最多展示 2 到 3 行
- 超出部分可截断
- 若为空，显示引导性占位文案，例如“写一段描述，让这个工作区更容易识别”

##### task 操作区

展示内容：

- `+` 新建 task 按钮
- task 列表标题
- task 数量

交互要求：

- 新建 task 按钮应当是左栏最明确的操作按钮
- 点击后打开创建 task 的弹层或内联表单

##### task 列表区

每个 task item 建议展示：

- task 标题
- task 简短描述或摘要
- task 状态
- 最后更新时间

task item 的展示规则：

- 当前选中的 task 要有高亮背景或描边
- 鼠标悬停时要有轻微反馈
- 描述最多展示 1 到 2 行
- 状态可以通过颜色或小圆点区分

建议状态映射：

- `idle`：灰色
- `running`：蓝色或绿色
- `failed`：红色
- `completed`：较弱的完成态颜色

task 列表的空态：

- 当没有 task 时，展示空态卡片
- 空态文案要明确引导用户点击 `+` 创建第一个 task

##### 当前用户信息区

展示内容：

- 用户头像
- 用户名
- 邮箱或简短身份信息

展示要求：

- 放在左栏底部
- 与上方 task 区域做视觉分隔
- 后续可以在这里扩展退出登录，但 v0.1 可先只展示

#### 3.3.2 右侧主区域

右侧在 v0.1 先承担空态和引导作用，不承载完整聊天。

建议展示内容：

- 欢迎标题
- 当前 workspace 简要说明
- 引导用户创建 task 或选择已有 task
- 若已有 task，可展示“最近 task”提示卡片

建议空态文案方向：

- 强调“从一个任务开始协作”
- 强调“每个 task 都会自动分配主 Agent 银河”

可选展示模块：

- 最近创建的 task
- 一个简单的使用步骤说明

建议的 3 步说明：

1. 创建 task
2. 进入 task
3. 和银河开始对话

#### 3.3.3 Workspace 编辑交互

编辑 `name` 和 `description` 的方式建议二选一：

- 方案 A：弹层编辑
- 方案 B：右侧面板编辑

v0.1 更建议：

- 使用弹层编辑

原因：

- 实现成本低
- 不破坏首页主布局
- 更适合最小版本

编辑弹层建议包含：

- 标题：编辑工作区
- 名称输入框
- 描述文本域
- 取消按钮
- 保存按钮

#### 3.3.4 新建 Task 交互

新建 task 的交互建议也使用弹层。

弹层建议包含：

- 标题：新建任务
- task title 输入框
- task description 文本域
- 取消按钮
- 创建按钮

创建成功后的页面行为建议：

- 自动关闭弹层
- 左侧列表新增这个 task
- 自动高亮新建 task
- 自动跳转到该 task 详情页

接口：

- `GET /api/workspace`
- `PATCH /api/workspace`
- `GET /api/tasks`
- `POST /api/tasks`

### 3.4 Task 详情页

目标：承接一个 task 下的主 Agent 对话。

建议继续沿用三段式结构，但保持最简：

- 左栏：task 列表
- 中栏：Agent 会话列表
- 右栏：聊天内容区

页面定位：

- 这是 v0.1 的核心工作页面
- 页面重点不是“联系人聊天”，而是“围绕某个 task 与主 Agent 持续协作”
- 即使当前只有一个 Agent，也要保留后续扩展成多 Agent 工作台的结构

#### 3.4.1 顶部任务栏

建议放在页面顶部，横跨主内容区。

展示内容：

- task 标题
- task 描述
- task 状态
- 返回 workspace 的入口

可选操作：

- 标记完成
- 更多操作占位按钮

展示要求：

- task 标题是详情页第一视觉层级
- task 描述作为辅助信息展示
- 状态标签要清晰可见

#### 3.4.2 左栏：Task 导航区

作用：

- 快速切换不同 task
- 让用户始终知道自己处于哪个任务上下文

展示内容：

- task 列表
- 当前 task 高亮
- 新建 task 快捷入口

展示规则：

- 与首页左栏风格保持一致
- 当前 task 的高亮要明显于其他项
- 列表滚动时保持顶部或底部关键操作可见

#### 3.4.3 中栏：Agent 会话区

v0.1 虽然只有一个 `银河`，但这里仍保留为独立中栏。

原因：

- 后续可直接扩展多个 Agent
- 当前也可以在视觉上表达“这是当前参与协作的 Agent”

中栏建议包含：

- 区域标题，例如“参与协作”
- 一个默认 Agent item：`银河`
- Agent 简短角色说明，例如“主 Agent”
- 当前状态标签，例如“在线”或“可用”

`银河` 卡片建议展示：

- 头像占位
- 名称：银河
- 角色：主 Agent
- 简短说明：负责响应当前 task 的主要问题

选中逻辑：

- v0.1 默认只有一个选中项
- 视觉上依然保留选中态，便于后续扩展

空态说明：

- v0.1 不需要空态
- 创建 task 后默认自动生成该 Agent

#### 3.4.4 右栏：聊天内容区

这是页面中信息密度最高的区域。

建议拆成 4 个垂直区块：

1. 会话头部
2. 消息列表
3. 错误或状态提示区
4. 底部输入区

##### 会话头部

展示内容：

- 当前 Agent 名称：银河
- 当前 task 标题
- 当前 task 简短描述

可选辅助信息：

- 本次对话围绕当前 task 持续进行

##### 消息列表

消息列表需要支持最基础的聊天体验。

用户消息展示建议：

- 靠右排列
- 使用较强的主题色或浅色填充区分
- 文本内容优先保证可读性

Agent 消息展示建议：

- 靠左排列
- 和用户消息形成明显视觉差异
- 支持长文本换行
- 支持 Markdown 基础渲染
- 支持 code block 的独立容器显示

系统消息展示建议：

- 居中或弱化展示
- 用于说明错误、执行失败、空结果等系统状态

消息间距建议：

- 单条消息之间保留稳定垂直间距
- 同一发送方连续消息可适当压缩间距

代码块展示建议：

- 单独背景容器
- 等宽字体
- 保留复制按钮位置，但 v0.1 可以先只做静态样式
- 超长代码支持横向滚动

时间信息建议：

- 每条消息可展示简短时间
- 时间可以弱化展示在气泡底部或外侧

##### 错误或状态提示区

在输入区上方保留一条轻量状态区域。

用于展示：

- 正在发送
- Agent 正在生成回复
- 本次调用失败
- 请重试

要求：

- 状态提示不能打断消息主流
- 失败信息要足够明确

##### 底部输入区

输入区建议固定在底部。

包含：

- 多行文本输入框
- 发送按钮

交互要求：

- 支持回车发送，`Shift + Enter` 换行，或反过来二选一，但要固定
- 发送中禁用重复提交
- 输入为空时发送按钮禁用

输入框展示要求：

- 默认至少显示 2 到 3 行高度
- 文本过长时内部滚动
- 占位文案要体现 task 语境，例如“围绕当前任务继续提问”

#### 3.4.5 页面状态

task 详情页至少需要覆盖这些展示状态：

##### 状态 1：初次进入且无消息

展示：

- 一段欢迎文案
- 当前 task 标题和描述
- 引导用户向银河发出第一条消息

##### 状态 2：已有历史消息

展示：

- 正常消息列表
- 自动滚动到底部

##### 状态 3：消息发送中

展示：

- 用户消息可先进入列表
- 状态区显示“银河正在思考”

##### 状态 4：回复成功

展示：

- assistant 消息进入列表
- task 状态可更新为 `running`

##### 状态 5：回复失败

展示：

- 状态区显示失败提示
- 可插入一条 `system` 消息
- 输入框仍可继续输入

#### 3.4.6 视觉层级建议

task 详情页的视觉重点顺序建议是：

1. 当前 task
2. 当前 Agent
3. 消息流
4. 输入动作

不要把界面做成所有元素同等强调，否则用户会失去“现在正在围绕哪个 task 协作”的感知。

接口：

- `GET /api/tasks/:taskId`
- `GET /api/tasks/:taskId/conversations`
- `GET /api/tasks/:taskId/messages`
- `POST /api/tasks/:taskId/messages`

## 4. 核心对象

为了避免 v0.1 代码阶段反复改数据结构，先统一最小对象定义。

### 4.1 User

```ts
type User = {
  id: string;
  username: string;
  email: string;
  avatarUrl: string | null;
  createdAt: string;
};
```

### 4.2 Workspace

```ts
type Workspace = {
  id: string;
  userId: string;
  name: string;
  description: string;
  createdAt: string;
  updatedAt: string;
};
```

### 4.3 Task

```ts
type Task = {
  id: string;
  workspaceId: string;
  title: string;
  description: string;
  status: "idle" | "running" | "failed" | "completed";
  createdAt: string;
  updatedAt: string;
};
```

说明：

- `idle`：刚创建，还没有开始对话
- `running`：用户已发起对话，当前等待 Agent 回复或最近一次执行成功
- `failed`：最近一次 Agent 执行失败
- `completed`：当前 task 已被用户视为完成

### 4.4 Conversation

```ts
type Conversation = {
  id: string;
  taskId: string;
  agentName: string;
  agentType: "main";
  createdAt: string;
};
```

说明：

- v0.1 只允许 `agentType: "main"`
- `agentName` 默认为 `银河`

### 4.5 Message

```ts
type Message = {
  id: string;
  taskId: string;
  conversationId: string;
  role: "user" | "assistant" | "system";
  content: string;
  status: "sending" | "success" | "failed";
  createdAt: string;
};
```

说明：

- `user` 表示用户发送的消息
- `assistant` 表示 `银河` 返回的消息
- `system` 用于保留系统提示或错误提示

请求体：

```json
{
  "username": "xinyang",
  "email": "xinyang@example.com",

}
```

```json
{
  "user": {
    "id": "user_001",

    "email": "xinyang@example.com",
    "avatarUrl": null,
    "createdAt": "2026-05-25T10:00:00.000Z"
  }
}
```

### `POST /api/login`

请求体：

```json
{
  "email": "xinyang@example.com",
  "password": "123456"
}
```

返回体：

```json
{
  "token": "mock-jwt-token",
  "user": {
    "id": "user_001",
    "username": "xinyang",
    "email": "xinyang@example.com",
    "avatarUrl": null,

  }
```


### 5.3 Task 接口

### `GET /api/tasks`

返回当前 workspace 下的 task 列表。

### `POST /api/tasks`

请求体：

```json
{
  "title": "写一个 SQL 学习任务",
  "description": "希望和主 Agent 对 SQL 基础概念进行持续交流"
}
```

返回体要求：

- 创建 task 成功
- 自动创建默认会话 `银河`
- 返回 task 基础信息

### `GET /api/tasks/:taskId`

返回单个 task 详情。

### `GET /api/tasks/:taskId/conversations`

返回当前 task 下的会话列表。

v0.1 预期固定返回：

```json
[
  {
    "id": "conv_001",
    "taskId": "task_001",
    "agentName": "银河",
    "agentType": "main",
    "createdAt": "2026-05-25T10:00:00.000Z"
  }
]
```

### 5.4 Message 接口

### `GET /api/tasks/:taskId/messages`

返回当前 task 的消息列表，按时间正序排列。

### `POST /api/tasks/:taskId/messages`

请求体：

```json
{
  "content": "什么是 SQL？"
}
```

后端处理步骤：

1. 先写入一条用户消息
2. 调用 `Claude Code CLI`
3. 写入一条 Agent 回复消息
4. 返回本次新增的消息结果

建议返回体：

```json
{
  "userMessage": {
    "id": "msg_user_001",
    "taskId": "task_001",
    "conversationId": "conv_001",
    "role": "user",
    "content": "什么是 SQL？",
    "status": "success",
    "createdAt": "2026-05-25T10:00:00.000Z"
  },
  "assistantMessage": {
    "id": "msg_assistant_001",
    "taskId": "task_001",
    "conversationId": "conv_001",
    "role": "assistant",
    "content": "SQL 是一种用于管理和查询关系型数据库的语言。",
    "status": "success",
    "createdAt": "2026-05-25T10:00:02.000Z"
  }
}
```

## 6. Agent 调用链路

v0.1 的主 Agent 暂时不是完整 Orchestrator，而是一个最小可运行的 CLI Adapter。

执行逻辑如下：

1. 前端调用 `POST /api/tasks/:taskId/messages`
2. 后端保存用户消息
3. 后端读取当前 task 的历史消息
4. 后端将历史消息整理成 prompt
5. 后端通过 `subprocess.run` 调用 `claude -p`
6. 获取标准输出
7. 保存 assistant 消息
8. 返回给前端渲染

参考调用方式：

```python
import subprocess

result = subprocess.run(
    ["claude", "-p", "what is SQL?"],
    capture_output=True,
    text=True
)

print(result.stdout)
```

在实际实现时，`what is SQL?` 应替换为拼装后的真实 prompt，而不是只传最后一条用户消息。

## 7. Prompt 拼装建议

虽然 v0.1 很简单，但如果只把最后一条消息直接发给 CLI，很快就无法继续多轮对话，所以需要保留最小上下文能力。

建议拼装方式：

1. 固定 system prompt
2. 追加 task 标题
3. 追加 task 描述
4. 追加最近 N 条历史消息
5. 最后追加本轮用户输入

建议 system prompt 基线：

- 你是 AgentHub 中的主 Agent 银河
- 你当前服务于一个 task 上下文
- 你需要围绕 task 目标持续回答用户
- 输出以清晰、直接、可继续追问为目标
- 如果涉及代码，尽量用 Markdown code block 输出

## 8. 错误处理

v0.1 至少需要覆盖下面几类错误：

### 8.1 认证错误

- 注册参数缺失
- 邮箱重复
- 登录密码错误
- token 无效

### 8.2 Task 错误

- task 不存在
- task 不属于当前用户
- task 标题为空

### 8.3 Agent 调用错误

- 本地机器未安装 `claude`
- `claude` 命令执行失败
- CLI 返回为空
- CLI 超时

建议处理方式：

- 后端返回明确错误码和错误信息
- 将本次 assistant 消息写为 `failed` 或补一条 `system` 消息说明失败原因
- task 状态更新为 `failed`

## 9. 数据持久化建议

v0.1 即使先不做复杂数据库设计，也建议至少保留下列持久化对象：

- users
- workspaces
- tasks
- conversations
- messages

原因：

- workspace 信息需要二次编辑
- task 列表需要长期保存
- 消息上下文需要支持多轮对话
- 后续升级为多 Agent 时，这套基础结构可以继续复用

## 10. UI 补充要求

- 前端整体风格保持简约、浅色系
- 能用图标表达的操作不强行叠加大量文字
- 页面文案使用中文
- task 列表、会话列表、聊天区层级要清晰
- `+` 新建 task 按钮需要足够显眼
- 为后续多 Agent 扩展预留中栏结构，不要把页面直接写死成单栏聊天页

## 11. 推荐开发顺序

建议按下面顺序推进：

1. 注册 / 登录
2. workspace 获取与编辑
3. task 创建与列表
4. task 详情页静态布局
5. message 表与消息接口
6. `Claude Code CLI` 调用封装
7. 前端发送消息并展示回复
8. 错误处理与状态修正

## 12. v0.1 完成标准

满足以下条件即可认为 v0.1 跑通：

1. 新用户可以注册并登录
2. 用户可以填写和修改 workspace 信息
3. 用户可以创建多个 task
4. 每个 task 自动拥有一个名为 `银河` 的主 Agent 会话
5. 用户进入任意 task 后可以发送消息
6. 后端可以成功调用 `Claude Code CLI`
7. Agent 返回内容可以持久化并显示在聊天区域
8. 页面刷新后，历史 task 和历史消息仍然存在

## 13. 与后续版本的关系

v0.1 是后续版本的基础，不是最终形态。

后续版本可在此基础上继续扩展：

- 多 Agent 群聊
- 主 Agent 拆解任务
- 文档 / PPT 产物区
- 协作记录
- Agent rules
- 文件与网页产物
- 流式输出

但在开始这些扩展之前，v0.1 要先把 `workspace -> task -> 主 Agent 对话 -> 回复落库 -> UI 展示` 这条链路打稳。
