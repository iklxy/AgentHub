### AgentHub v0.2 开发文档

- 日期：2026-05-25
- 文档目标：将 `v0.2` 的会话模型、Agent Harness、前后端交互与落地边界补成可直接进入开发的详细规格。
- 当前版本基线：`v0.1` 已完成 `workspace -> task -> 主 Agent 银河 -> Claude CLI 返回 -> 消息展示` 的最小闭环。

## 1. v0.2 定位

v0.2 的核心目标不是继续扩充更多页面，而是把 AgentHub 从“每轮都手工拼接历史上下文再调用 Agent”的模式，升级为“基于外部 Agent 原生 session 能力持续对话”的模式。

这意味着 `v0.2` 要解决的核心问题有三件：

1. 将 `task_sessions.id` 直接作为 `Claude Code CLI` 的会话标识使用
2. 让同一个 session 的连续对话通过 `resume` 能力继续执行，而不是由 AgentHub 手工查历史再回灌
3. 在一个 `task` 内支持创建多个 session，并在前端明确区分“主 Agent 默认 session”和“新增 session”

这一版的核心不是完整多 Agent 编排，而是先把“多 session + 原生会话续写 + Agent 工作目录”这一层打稳。

## 2. v0.2 范围

### 2.1 必做

- 为主 Agent `Galaxy / 银河` 接入 Claude CLI 的原生 `session_id`
- 使用 `task_sessions.id` 作为 Claude CLI 会话标识
- 在单个 `task` 下支持创建多个 session
- 新增 session 创建接口 `POST /api/sessions`
- 前端 task 页面支持：
  - 展示 session 列表
  - 区分默认主 session 和新增 session
  - 点击切换当前 session
  - 新建 session 弹窗
- 后端消息发送逻辑改为“按 session 调用 Agent”
- Agent 运行目录切换到 `.AgentHub` harness 结构
- `Agent.md` 规则文件纳入 Agent 运行目录
- 日志中明确记录：
  - 当前 `task_id`
  - 当前 `session_id`
  - 当前 `agent_id`
  - Claude CLI 执行命令模式是 `start` 还是 `resume`

### 2.2 暂不做

- skill 系统
- Claude memory 文件解析与治理
- Aries 的真实执行适配
- 群聊模式自动分派
- artifacts 最终目录方案定稿
- rules 可视化编辑页
- session pin / archive / search
- 主 Agent 自动创建子 session 的完整编排闭环

### 2.3 版本判断

`v0.2` 仍然不是完整多 Agent 协作平台。

它解决的是：

- 会话如何真正连续
- 一个 task 下如何有多个可切换对话
- Agent 的运行目录如何从“单次 subprocess”升级为“带本地 harness 的持续运行上下文”

## 3. 当前问题与升级动机

### 3.1 v0.1 的问题

当前 `v0.1` 的主 Agent 调用链路是：

1. 用户发送消息
2. 后端从 `messages` 表取这个 task 下的历史消息
3. 手工拼接成 `history`
4. 把 `task + history + 当前输入` 传给 Python agent
5. Python agent 再组 prompt 调用 Claude CLI

这个方案能跑通，但有明显限制：

- 上下文窗口完全由 AgentHub 手工控制
- 当前实现只取最近几条消息，不是真实原生会话续写
- 历史消息和 Agent 的内部会话状态是割裂的
- 后续要做多 session、多 Agent 时，会话管理复杂度会快速上升

### 3.2 v0.2 的升级思路

Claude CLI 已支持原生 session 续写能力，因此 `v0.2` 应把会话管理的“上下文续写”职责尽量下放给 Claude 本身。

AgentHub 在 `v0.2` 中主要负责：

- 生成并持有稳定的 `task_sessions.id`
- 维护“哪个 task 下有哪些 session”
- 记录消息、审计和页面状态
- 选择应该把哪条用户输入发给哪个 Agent / 哪个 session

AgentHub 不再负责：

- 每次都把完整历史消息重新拼接后喂给 Claude

## 4. 会话模型定义

### 4.1 核心对象

`v0.2` 中涉及四个不同层次的对象，必须区分：

1. `task`
2. `task_session`
3. `agent`
4. `claude runtime session`

它们的关系如下：

- 一个 `task` 下可以有多个 `task_session`
- 一个 `task_session` 在 `v0.2` 只绑定一个 `primary_agent`
- 对于 `Claude Code CLI` 类型的 agent，一个 `task_session.id` 对应一个 Claude 原生 session

### 4.2 默认主 session

每个 task 在创建时默认生成一个主 session，建议命名：

- `title`: `主对话`
- `chat_mode`: `single`
- `primary_agent_id`: `Galaxy`
- `session_kind`: `primary`

这个默认 session 是 task 的主入口。

### 4.3 新增 session

在 `v0.2` 中，用户可以在一个 task 内再创建额外 session。

新增 session 的典型用途：

- 围绕同一 task 开一个新的讨论分支
- 让另一个 Agent 处理该 task 的某个子问题
- 后续支持由主 Agent 主动创建“子会话”

### 4.4 前端必须区分的两类 session

前端展示上必须明确区分：

1. 默认主 session
2. 后续新增 session

推荐展示规则：

- 默认主 session 固定显示在 session 列表最上方
- 默认主 session 标签：`主会话`
- 新增 session 标签：`新增会话`
- 如果后续是由主 Agent 自动创建，可再补充：`由银河创建`

## 5. Claude CLI 会话接入方案

### 5.1 会话标识

`v0.2` 统一使用：

- `task_sessions.id`

作为 Claude CLI 的会话标识。

即：

- 首次启动：使用 `--session-id <task_session_id>`
- 后续续写：使用 `--resume <task_session_id>`

### 5.2 首次消息

当某个 session 第一次收到用户消息时：

1. 后端确认该 session 还未启动过 Claude 原生会话
2. 通过 `claude --session-id "<task_session_id>" -p "<message>"` 发起对话
3. 将本轮用户消息和 assistant 消息照常写入数据库

### 5.3 后续消息

当某个 session 已经启动过后：

1. 后端不再手工查询历史消息拼接 prompt
2. 直接调用：
   - `claude -p --resume "<task_session_id>" "<message>"`
3. 数据库仍然记录消息，但只作为系统记录层，不再作为 Claude 上下文拼装来源

### 5.4 数据库与 Claude session 的关系

数据库中的 `messages` 在 `v0.2` 的职责是：

- 页面消息展示
- 审计与排错
- 后续规则/产物/步骤联动

Claude session 的职责是：

- 保存对话上下文
- 决定连续多轮中的隐式状态

结论：

- 数据库是系统记录源
- Claude session 是运行时上下文源

## 6. Agent Harness 目录结构

## 6.1 根目录

项目根目录下新增：

```text
.AgentHub/
├── .Galaxy/
├── .Aries/
├── skills/
│   ├── skill1/
│   │   └── skill.md
│   └── skill2/
│       └── skill.md
└── runtime/
    └── logs/
```

说明：

- `.AgentHub` 是整个 AgentHub 本地 Agent 运行时根目录
- 每个 Agent 一个独立目录，命名规则是 `.` + AgentName
- `skills/` 对所有 Agent 可见
- `runtime/logs/` 用于保留 AgentHub 自己的运行日志

### 6.2 单 Agent 目录

以 `.Galaxy` 为例：

```text
.AgentHub/.Galaxy/
├── Agent.md
├── sessions/
│   ├── <task_session_id_1>/
│   │   ├── meta.json
│   │   └── logs/
│   └── <task_session_id_2>/
│       ├── meta.json
│       └── logs/
└── workspace/
```

说明：

- `Agent.md`：该 Agent 的本地规则文件，由 AgentHub 写入或维护
- `sessions/<task_session_id>/`：
  - 不是 Claude 原生 session 的权威来源
  - 而是 AgentHub 自己保留的会话运行元数据目录
- `workspace/`：该 Agent 执行时的工作目录

### 6.3 关于 session 目录的原则

要明确：

- Claude session 的真正上下文由 Claude 自己维护
- `.AgentHub/.Galaxy/sessions/<task_session_id>/` 仅保存：
  - 本地元数据
  - 调用日志
  - 未来可能的本地运行附加文件

因此不要在 `v0.2` 再尝试自己做一套完整的历史消息重放系统。

### 6.4 skills 处理原则

`v0.2` 不处理 skill 功能。

仅保留目录约定，不做：

- skill 注入
- skill 选择
- skill UI
- skill API

- `Aries` 预留 `Agent.md`
- 启动 Agent 时，运行目录中必须能看到对应 `Agent.md`

### 7.3 暂不做

## 8. 数据库与迁移调整

### 8.1 当前可复用表

当前数据库已有这些关键表，可直接复用：

- `agents`
- `tasks`
- `task_sessions`
- `messages`
- `session_participants`
- `audit_events`

### 8.2 建议新增字段

为了让 `v0.2` 的 session 行为可控，建议对 `task_sessions` 增加以下字段：

```sql
ALTER TABLE task_sessions
ADD COLUMN session_kind VARCHAR(32) NOT NULL DEFAULT 'primary',
ADD COLUMN created_from_session_id UUID NULL REFERENCES task_sessions(id) ON DELETE SET NULL,
ADD COLUMN runtime_provider VARCHAR(64) NULL,
ADD COLUMN runtime_session_key TEXT NULL,
ADD COLUMN started_at TIMESTAMPTZ NULL;
```

字段说明：

- `session_kind`
  - `primary` / `branch`
  - 区分默认主 session 和新增 session
- `created_from_session_id`
  - 记录该 session 是否由另一个 session 派生
- `runtime_provider`
  - 当前运行时提供方，例如 `claude_code`
- `runtime_session_key`
  - 保存真正传给运行时的 session key
  - `v0.2` 中可直接等于 `task_sessions.id`
- `started_at`
  - 标记该 session 是否已经完成第一次 runtime 启动

### 8.3 为什么需要这些字段

虽然 `task_sessions.id` 已经存在，但如果不补这些字段，系统很难明确判断：

- 这是默认主 session 还是新增 session
- 该 session 是否已经启动过 Claude runtime
- 该 session 是否由另一个 session 创建

这些信息不能只靠前端临时推断。

### 8.4 messages 表使用原则

`messages` 表在 `v0.2` 仍然保留完整记录，但使用原则改变：

- 前端展示：继续依赖数据库
- Agent 上下文注入：不再默认依赖数据库回放

### 8.5 迁移影响

这是一次结构升级，但不需要推翻现有表。

迁移影响如下：

- `task_sessions` 新增字段
- 旧 task 的默认 session 需要回填：
  - `session_kind = 'primary'`
  - `runtime_provider = 'claude_code'`
  - `runtime_session_key = id`

## 9. 后端接口设计

### 9.1 保留接口

这些接口继续保留：

- `GET /api/tasks/{taskId}`
- `GET /api/tasks/{taskId}/messages`
- `POST /api/tasks/{taskId}/messages`

但 `POST /api/tasks/{taskId}/messages` 的内部逻辑要改为“按当前 session 发消息”。

### 9.2 新增 session 接口

#### `GET /api/tasks/{taskId}/sessions`

用途：

- 获取某个 task 下的 session 列表

返回字段建议：

```json
[
  {
    "id": "uuid",
    "taskId": "uuid",
    "title": "主对话",
    "chatMode": "single",
    "sessionKind": "primary",
    "primaryAgentId": "uuid",
    "primaryAgentName": "Galaxy",
    "startedAt": "2026-05-25T10:00:00Z",
    "lastActiveAt": "2026-05-25T10:10:00Z",
    "lastMessagePreview": "最近一次回复摘要"
  }
]
```

#### `POST /api/sessions`

用途：

- 在单个 task 下创建新的 session

请求体：

```json
{
  "taskId": "uuid",
  "title": "文档整理分支",
  "chatMode": "single",
  "primaryAgentId": "uuid"
}
```

约束：

- `taskId` 必填
- `title` 必填
- `chatMode` 当前仅允许 `single`
- `primaryAgentId` 当前仅允许：
  - `Aries`
  - 后续其他非默认 Agent
- `Galaxy` 不允许通过这个弹窗手工再选一次作为新增 session 的目标 Agent

原因：

- `Galaxy` 已经是每个 task 自带的默认主 session
- 新增 session 的目的，是补充分支会话或预留其他 Agent

创建逻辑：

1. 校验用户是否有该 task 权限
2. 校验 `primaryAgentId` 是否存在且启用
3. 新建 `task_sessions`
4. 写入：
   - `session_kind = 'branch'`
   - `runtime_provider = 'claude_code'` 或预留 provider
   - `runtime_session_key = task_sessions.id`
5. 返回新建 session

#### `GET /api/sessions/{sessionId}`

用途：

- 获取单个 session 详情

#### `PATCH /api/sessions/{sessionId}`

用途：

- 更新 session 标题
- 当前不开放 `chat_mode` 变更
- 当前不开放 `primary_agent_id` 变更

### 9.3 消息发送接口改造

#### `POST /api/tasks/{taskId}/messages`

建议改为：

```json
{
  "sessionId": "uuid",
  "content": "用户输入"
}
```

原因：

`v0.2` 已经不再是“task 下只有一个主对话”，所以发送消息时必须显式带上 `sessionId`。

后端执行步骤：

1. 校验 `taskId` 与 `sessionId` 是否匹配
2. 查询该 session 的 `primary_agent_id`
3. 根据 agent 类型决定调用哪个 adapter
4. 如果 session 未启动：
   - `claude --session-id "<session_id>" -p "<content>"`
5. 如果 session 已启动：
   - `claude -p --resume "<session_id>" "<content>"`
6. 写入数据库消息
7. 更新 `task_sessions.last_active_at`
8. 更新 `tasks.current_session_id`

### 9.4 为什么仍然保留消息接口在 task 下面

因为当前页面结构仍然是“进入 task 后在内部切 session”。

所以：

- `task` 仍然是页面容器
- `session` 是 task 内的对话容器

这种路径结构和当前 UI 更一致。

## 10. Agent 执行层改造

### 10.1 当前 v0.1 的问题

当前 Python agent 会接收：

- task title
- task description
- history-json
- user-input

这套参数模型在 `v0.2` 需要调整。

### 10.2 v0.2 的执行输入

对 `Claude Code CLI` 型 Agent，Python adapter 不再默认接收完整历史消息，而是接收：

- `agent_name`
- `session_id`
- `agent_workdir`
- `agent_rule_file`
- `user_input`
- `is_first_message`

### 10.3 首次与续写的执行分支

首次消息：

- 使用 `--session-id`

续写消息：

- 使用 `--resume`

### 10.4 建议保留的 prompt 注入

虽然不再手工灌历史，但首次启动时仍建议把这些结构注入到首轮输入中：

- task title
- task description
- 当前 session title
- Agent.md 规则摘要

这样 Claude 在第一次建立 session 时就知道：

- 当前任务是什么
- 当前这个分支会话想解决什么

### 10.5 后续消息原则

后续消息只传用户本轮输入，不再拼数据库历史。

## 11. 前端页面改造

### 11.1 task 页整体结构

`v0.2` 的 task 工作台应改成三栏：

1. 左栏：task 与 session 导航
2. 中栏：Agent 列
3. 右栏：当前 session 的消息区

### 11.2 左栏 session 区

左栏除了 task 基本信息，还需要新增 session 区块。

建议展示：

- 区块标题：`Sessions`
- `+` 新建 session 按钮
- session 列表

每个 session item 展示：

- title
- 标签：
  - `主会话`
  - `新增会话`
- primary agent name
- last active time

交互要求：

- 点击某个 session，右侧消息区切换到该 session
- 当前激活 session 要有高亮态

### 11.3 新建 session 弹窗

点击新建 session 后弹窗。

表单字段：

- `session.title`
- `session.chat_mode`
- `session.primary_agent_id`

当前前端交互约束：

- `chat_mode` 先只显示 `single`
- `Galaxy` 不出现在可选列表里
- `Aries` 出现在列表里，但可以在选择后提示“当前先完成 session 创建，真实调用将在后续版本接入”

### 11.4 中栏 Agent 区

中栏展示当前 session 的主 Agent 信息。

对于默认主 session：

- 显示 `Galaxy`

对于新增 session：

- 显示该 session 对应的 `primary_agent`

### 11.5 右栏消息区

右栏顶部应增加当前 session 信息：

- session 标题
- session 标签
- primary agent 名称

这样用户能明确知道“当前正在和哪个 session 对话”。

## 13. v0.2 与 v0.1 的核心差异

### 13.1 上下文来源变化

`v0.1`：

- 数据库历史消息是 Agent 上下文的主要来源

`v0.2`：

- Claude 原生 session 是 Agent 上下文的主要来源
- 数据库消息只负责记录和展示

### 13.2 对话结构变化

`v0.1`：

- 一个 task 默认只有一个会话视角

`v0.2`：

- 一个 task 下可以有多个 session

### 13.3 Agent 运行形态变化

`v0.1`：

- 单次 subprocess 调用

`v0.2`：

- 基于固定 session id 的连续运行上下文

- 核对数据库消息记录是否完整

## 16. 验收标准

`v0.2` 完成时，至少要满足：

1. 创建 task 后自动生成一个默认主 session
2. 在 task 页面可以看到 session 列表
3. 点击“新建 session”可以创建新的 session
4. 给某个 session 发第一条消息时，系统走 `--session-id`
5. 给同一个 session 发第二条消息时，系统走 `--resume`
6. 切换到另一个 session 发消息时，系统会启用另一个独立 session
7. 数据库里的消息仍然完整记录每轮 user / assistant 消息
8. 日志里可以清楚看到当前命中的是哪个 `task / session / agent`

## 17. 本版明确不做的错误方向

为了避免开发过程中重新回到 `v0.1` 逻辑，`v0.2` 明确禁止以下实现方向：

- 继续默认按 task 全量查数据库历史再喂给 Claude
- 不区分默认主 session 与新增 session
- 新建 session 但消息发送时仍然不带 `sessionId`
- 在没有本地 harness 目录的情况下继续裸调用 Agent
- 把 skill 系统混进本轮开发

## 18. 一句话总结

`v0.2` 的本质不是“多加一个 session 列表 UI”，而是把 AgentHub 的对话运行时从“手工拼上下文”升级为“基于真实 Agent session 的持续对话系统”。
